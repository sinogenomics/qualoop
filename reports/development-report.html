在 Cursor 里怎么说（业务项目专用）

专业做法（推荐）：一次配置，日常一句。见 PROFESSIONAL_SETUP.md。

配置一次：运行 scripts/install-cursor-rule.ps1，或将规则复制到业务项目 .cursor/rules/； 可选把本方法论文件夹添加到 Cursor 工作区（多根工作区）。
日常只说：Qualoop 检查（首次说 Qualoop 初始化）。

下面长话术仅作备查，不必每次复制。工程清单：ADOPTION_GUIDE.md。

  • 用 Cursor 打开业务项目文件夹
  • 可选：复制 templates/qualoop.cursor.rule.mdc → 业务项目 .cursor/rules/qualoop.mdc

方法论路径（话术里请改成你的）：e:/path/to/qualoop

话术 2 · 每轮检查（最常用）

复制整段到 Cursor 聊天框 
按 Qualoop 在本项目跑【一轮】质量检查：

@tools/qualoop/METHODOLOGY.md §1.3–§1.5

必须：至少 1 条修改意见；每条 goal_alignment_note；Scorer 打分 value_score，≥60 为合格；
当轮至少 1 条 value_qualified，否则加深检查直到有。禁止「没问题」式交白卷。

输出 automation/issues.json 与 reports/latest_issues.md，并列出合格数、最高分。

话术 3 · 低分 / 空轮

复制整段到 Cursor 聊天框 
上一轮 Qualoop 不合格：高分意见不足。请加深检查直到至少 1 条 value_score≥60 且对齐 North Star。
参考 @tools/qualoop/METHODOLOGY.md §1.4.2。不要宣布「系统健康无需修改」。

话术 4 · 只改高分 Issue（慎用）

复制整段到 Cursor 聊天框 
仅处理 automation/issues.json 里 value_qualified=true 且已分配的 Issue。
@tools/qualoop/METHODOLOGY.md §2.4
改完后 verifier 复跑；无法验证则 resolved=false。

三条首要前提

同等不可妥协，合取决定「当轮检查是否合格」。

前提一 · 目标对齐（North Star）

修改意见须严谨、审慎，推动系统朝五条最终目标前进，不得背离（如关检测清零 backlog）。

前提二 · 每轮必有产出（Inspection depth）

每轮至少提出一条意见；提不出 = 深度不足。全绿时仍须产出对齐的 improvement，直至 Scorer 判定合格。

前提三 · 价值评分门槛（Scorer）

专门智能体对每条意见按贡献度打分并记录；分值低于 min_value_score 为不合格，须继续检查直至有足够高分合格意见。

合格一轮：当轮 value_qualified 数量 ≥ min_qualified_per_round（默认 ≥1），且每条通过对齐闸门。

五角色协作模型

Guardian
监督
Tester → Scorer → Scheduler → Executors
发现 → 评分 → 分派（仅合格）→ fixer / improver / verifier

Tester

多渠道 discovery，写入 Issue Store，产出候选意见。

Scorer

唯一写入 value_score*；判定 value_qualified；低分触发加深。

Scheduler

唯一分配写入者；仅分派通过 Scorer 评估的合格 Issue；path lock + lease。

Executors

有界执行：fixer、improver、verifier；策略表约束 auto-fix。

Guardian

子进程监督、退避重启、报告快照。

Issue Store

中心化台账，串联发现→评分→分派→验证全链路。
